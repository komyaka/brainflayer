# Prompt templates (Copilot Web → agent: orchestrator)

Ниже — три режима: **минимальный**, **средний**, **максимальный**.  
Выбирай по задаче: чем выше режим, тем больше проверки/документации/независимых ревью.

> Важно: ты выбираешь агента **orchestrator** в GitHub Copilot Web и вставляешь один из шаблонов.

---

## 1) Минимальный режим (FAST PATH)
Используй для небольших правок, когда нужна скорость: ≤1–2 файла, без изменений API/архитектуры, без новых зависимостей.

```text
РЕЖИМ: MIN (fast)

РЕПОЗИТОРИЙ:
[ссылка на репозиторий/ветку/PR или просто "текущий репозиторий"]

ЗАДАЧА:
[1–3 предложения: что нужно сделать]

КРИТЕРИИ ПРИЁМКИ:
- [pass/fail 1]
- [pass/fail 2]

ОГРАНИЧЕНИЯ:
- [что нельзя менять / совместимость / версии]

КОНТЕКСТ:
- [если баг: шаги воспроизведения + лог/stacktrace]
```

Ожидаемая цепочка: `coder → auditor`.

---

## 2) Средний режим (STANDARD)
Используй для типовых задач: новые фичи среднего размера, багфиксы с тестами, изменения поведения.

```text
РЕЖИМ: MID (standard)

РЕПОЗИТОРИЙ:
[ссылка/ветка/PR или "текущий репозиторий"]

ЗАДАЧА:
[что построить/пофиксить/изменить и зачем]

КРИТЕРИИ ПРИЁМКИ (измеримые):
- [pass/fail 1]
- [pass/fail 2]
- [pass/fail 3]

ОГРАНИЧЕНИЯ:
- [совместимость, версии, производительность, безопасность, запреты]

В РАМКАХ (IN SCOPE):
- [пути/модули, если известны]

ВНЕ РАМОК (OUT OF SCOPE):
- [что не трогать]

КОМАНДЫ ЗАПУСКА/ТЕСТОВ:
[если знаешь — перечисли; если нет — "unknown, derive from repo"]

КОНТЕКСТ:
- [ссылки Issue/PR, примеры входов/выходов, логи]
```

Ожидаемая цепочка: обычно `architect → coder → qa-test (по триггерам) → auditor`, плюс `security-review/performance-review/dx-ci/docs-writer` по триггерам.

---

## 3) Максимальный режим (HARDENED / SHIP READY)
Используй для “сделай готовое решение/программу”: крупные изменения, новые подсистемы, высокая цена ошибки, публичные API, безопасность/перф критичны.

```text
РЕЖИМ: MAX (hardened)

РЕПОЗИТОРИЙ:
[ссылка/ветка/PR или "текущий репозиторий"]

ЦЕЛЬ (что должно получиться):
[например: "готовый сайт", "CLI утилита", "сервис", "библиотека", "плагин", "расширение"]

ФУНКЦИОНАЛ (список требований):
- [требование 1]
- [требование 2]
- [требование 3]

КРИТЕРИИ ПРИЁМКИ (строго pass/fail):
- [тест/команда/проверка 1]
- [тест/команда/проверка 2]
- [тест/команда/проверка 3]
- [нефункциональные: perf/memory/latency/security]

ОГРАНИЧЕНИЯ И РИСКИ:
- [версии, окружение, деплой, совместимость]
- [ограничения по времени/ресурсам]
- [важные риски]

АРХИТЕКТУРНЫЕ ОГРАНИЧЕНИЯ:
- [например: "не менять публичные интерфейсы", "сохранить обратную совместимость", "не добавлять зависимости"]

В РАМКАХ (IN SCOPE):
- [модули/пакеты/сервисы/директории]

ВНЕ РАМОК (OUT OF SCOPE):
- [явные запреты]

КОМАНДЫ СБОРКИ/ТЕСТОВ/ЛИНТА:
[если знаешь — перечисли; если нет — "unknown, derive from repo"]

СЦЕНАРИИ (use-cases / примеры):
- вход → ожидаемый выход
- API примеры / CLI примеры
- edge cases

КОНТЕКСТ:
- ссылки Issue/PR
- логи/трейсы
- ограничения окружения
```

Ожидаемая цепочка: `architect → coder/refactor-maintainer → qa-test → security-review → performance-review → dx-ci → docs-writer → auditor` (строго по триггерам, без параллели).
